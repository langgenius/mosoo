You can tell a lot about a platform from the metrics on the home page. If the first chart is _tokens per day_, the platform is selling capacity. If the first chart is _tasks shipped_, it's selling outcomes.

We chose outcomes.

## The metrics we actually look at

| Metric | Why it matters |
|---|---|
| **Time-to-first-result** | New users either see the magic in 5 minutes or churn. |
| **Resolution rate** | The percentage of agent runs that end in a clean state — not a `?` from the user. |
| **Wall-clock saved** | Compared to the human baseline. The only ROI metric that survives a procurement review. |

Token counts are interesting for capacity planning, but they're not interesting for product. A 10× model that solves the problem in one call is cheaper, faster, and better than a 1× model that loops six times.

## What this means for the runtime

We optimize the runtime for **fewer, better calls**, not more, cheaper ones. That means investing in tooling that lets agents pause and ask for help, refuse work that's underspecified, and emit structured progress that humans can interrupt. It is, frankly, less fun than reading throughput charts. But it's what makes the platform feel like a colleague rather than a slot machine.
